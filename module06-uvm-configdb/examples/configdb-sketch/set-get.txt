UVM ConfigDB sketch (literacy — set / get / path)
===============================================

API pattern (offline UVM):
  uvm_config_db#(T)::set(cntxt, inst_name, field, value)
  uvm_config_db#(T)::get(cntxt, inst_name, field, value)

Teaching triple (browser lab model):
  set(path, field, value)
  get(from_path, field) → walk up ancestors of from_path

Starter example:
  set("env.agent", "vif", uart_vif)
  get("env.agent.drv", "vif") → HIT (walk: drv → agent → env …)

Lookup rules:
  1. Field name must match exactly
  2. Set path must be an ancestor of get path (or empty top)
  3. Among matches, longest path wins (most specific)

Examples:
  set env + set env.agent (same field) → agent child sees env.agent value
  set env.agent.vif but get is_active → MISS
  set on env.agent0, get from env.agent1.drv → MISS (siblings)

Typical uses:
  virtual interface handles to drivers/monitors
  is_active / is_passive mode
  timeouts, verbosity knobs scoped by path

Timing:
  set early (test or env build) before downstream get in build/connect

Common mistakes:
  typo in field string
  set too late (getter already ran)
  expecting sibling agent to see another agent's set
